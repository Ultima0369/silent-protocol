# ADR-0001：Silent Protocol 整体架构决策

## 状态

已接受

## 日期

2026-05-18

## 背景

现有智能体通信协议（MCP、A2A、CrewAI 等）存在以下未满足的需求：

1. 所有协议都假定智能体之间的通信应以**人类可读**为默认格式，导致 token 效率低下。
2. 没有协议原生支持**跨模型、跨供应商、跨云/本地边界**的异质协作。
3. 没有协议明确设计**人类仅在顶层尝菜**的反馈回路——现有系统要么全自动（人类在外），要么每步都要人确认（阻塞）。
4. 没有协议为**硅基实体之间的原生通信**（非自然语言）设计独立的通信层。

## 决策

采用 **Silent Protocol 四方协作架构** 作为核心设计模式。

### 架构概览

┌─────────────────────────────────────────────────────────────┐
│                     人类（产品经理）                         │
│                 自然语言 · 尝菜式反馈                        │
└────────────────────────┬────────────────────────────────────┘
                         │  Chatbox / API
                         ▼
┌─────────────────────────────────────────────────────────────┐
│              云端 DeepSeek（翻译官 + 任务拆解）               │
│             自然语言 ↔ 紧凑协议 双向翻译                      │
└────────────────────────┬────────────────────────────────────┘
                         │  neca MCP · 紧凑协议
                         ▼
┌─────────────────────────────────────────────────────────────┐
│              neca Server（通信网关 + 路由）                   │
│         compact-protocol → 路由到正确目标                     │
└───────┬─────────────────────────────────┬───────────────────┘
        │ 紧凑协议                         │ 紧凑协议
        ▼                                  ▼
┌──────────────────┐          ┌──────────────────────────┐
│  本地 Claude Code │          │  云端 Claude API          │
│  （总工程师）      │◄────────►│  （顾问团）               │
│   执行·读写·编译   │  求助     │   知识·推理·审查          │
└──────────────────┘          └──────────────────────────┘

### 核心设计原则

1. **紧凑优先**：硅基实体之间的通信使用预定义的、高密度的紧凑格式，不迁就人类可读性。
2. **分层解耦**：人类层、翻译层、执行层、知识层使用各自最合适的语言，通过明确定义的接口交互。
3. **异步非阻塞**：任务派发后，发送方不阻塞等待，接收方完成后异步汇报。
4. **尝菜式反馈**：人类只在关键节点介入，不参与过程噪音。
5. **向前兼容**：协议设计预留从 JSON 到二进制的升级路径。

## 考量

### 优势

- 将自然语言的歧义限制在人类-AI 接口层，硅基-硅基通信层保持零歧义。
- 大幅降低 token 消耗（预计减少 50-70% 的非有效载荷 token）。
- 四方异质协作覆盖了更广的知识和能力分布。
- 人类不阻塞在技术细节中。

### 劣势

- 架构复杂度高于单体 MCP server。
- 需要维护 neca 网关的路由和会话管理逻辑。
- 调试时需要额外的协议解析工具。

### 备选方案

1. **纯 MCP**：只用 MCP 协议，所有实体之间走自然语言。→ 拒绝：token 浪费严重。
2. **纯 A2A**：使用 Google A2A 标准。→ 拒绝：A2A 不设计"人类在顶层"的模式。
3. **单体 agent**：所有逻辑在单一 agent 内完成。→ 拒绝：失去了异质协作的多样性优势。

## 影响

- neca 需要新增 `protocol/` 和 `relay/` 模块。
- 现有的 `delegate` / `batch` 工具需要改造以支持紧凑协议。
- 需要定义一套完整的消息类型（见 ADR-0002）。
- 需要实现会话管理和超时重试机制。

## 遵守情况

所有后续 ADR 和代码实现都必须遵循本决策定义的四层架构和紧凑优先原则。
