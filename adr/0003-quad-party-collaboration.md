# ADR-0003：四方协作模式与尝菜回路

## 状态

已接受

## 日期

2026-05-18

## 背景

现有多智能体系统在人类参与度的设计上存在两个极端：

1. **全自动模式**（AutoGen、CrewAI 的某些模式）：人类设定目标后不再介入。优点是快，缺点是无法保证输出符合人类预期。
2. **逐级审批模式**（Human-in-the-Loop）：每一步都需要人类确认。优点是可控性高，缺点是人类的认知带宽成为瓶颈。

Silent Protocol 需要一个更适合真实人类认知特征的参与模式。

## 决策

采用**尝菜式反馈回路**。

### 模式定义

人类在协作中的角色类比为**产品经理/美食评论家**：

1. 人类设定目标（"我想吃湘菜"）。
2. 智能体们自行分工协作（采购、备菜、烹饪）。
3. 智能体们完成后，将成品端到人类面前。
4. 人类尝一口，给反馈（"咸了点，再加点醋"）。
5. 智能体们根据反馈修正，再端上来。
6. 重复 4-5 直到人类满意。

### 设计原则

| 原则 | 描述 |
|------|------|
| 不打断过程 | 人类不在智能体执行过程中被拉入技术细节 |
| 只反馈结果 | 人类只对最终/阶段性成果给出评价 |
| 异步非阻塞 | 智能体在执行时人类可以去做其他事 |
| 反馈即指令 | 人类的反馈自动转换为新的任务拆解 |

### 与现有模式的对比

| 维度 | HITL（人在回路） | 全自动 | 尝菜式（本协议） |
|------|-----------------|--------|-----------------|
| 人类参与频率 | 每步 | 仅初始 | 仅关键节点 |
| 人类认知负荷 | 高 | 无 | 低 |
| 过程可控性 | 极高 | 低 | 中高 |
| 执行速度 | 慢（人类成为瓶颈） | 快 | 快 |
| 人类成就感 | 疲惫 | 疏离 | 满足 |

## 考量

### 优势

- 尊重人类的认知带宽约束。
- 避免"生理性劫持"（疲劳时做出糟糕决策）。
- 让人类保持对最终成果的掌控感。
- 给智能体足够的自由空间探索最优方案。

### 劣势

- 如果人类反馈太模糊（"不好吃但我说不清哪里不好"），智能体可能需要多轮迭代。
- 需要智能体具备根据模糊反馈自行揣摩的能力（这对当前模型是已知挑战）。

## 影响

- 任务拆解层（云端 DeepSeek）需要学会将"自然语言反馈"转换为"可执行的任务修正"。
- neca 需要支持任务暂停、重试、覆盖等生命周期操作。
