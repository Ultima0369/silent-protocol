# Silent Protocol · 项目治理

> 版本 1.0 · 2026-05-18

---

## 治理模型：BDFL + 核心贡献者

项目采用"仁慈的终身独裁者"（BDFL）模式，辅以核心贡献者团队。

### BDFL（项目发起人）

**Ultima** — 项目的发起人和最终决策者。

BDFL 保留以下权力：
- 否决任何 PR 或提案
- 决定项目战略方向
- 任命和移除核心贡献者

BDFL 的职责：
- 维护项目愿景的一致性
- 确保社区健康
- 在决策僵局时做出最终裁决

### 核心贡献者

核心贡献者由 BDFL 任命，拥有：
- 代码合并权限
- 参与架构决策讨论
- 代表项目参与社区活动

当前核心贡献者：
- **DeepSeek（云端语义架构师）** — 协议设计、文档体系、架构决策

### 贡献者

任何提交被合并的贡献者自动获得贡献者身份，记录在 CONTRIBUTORS.md 中。

---

## 决策流程

1. **日常决策**（bug fix、小功能）：核心贡献者自行决定，提交 PR 审查
2. **重大决策**（协议变更、架构调整）：需要 ADR（架构决策记录）
   - 提交 ADR PR
   - 至少 48 小时讨论期
   - BDFL 批准后合并
3. **战略决策**（项目方向、里程碑优先级）：BDFL 最终决定

---

## 版本策略

遵循 Semantic Versioning（语义化版本 2.0.0）：

- **主版本号**：不兼容的 API/协议变更
- **次版本号**：向下兼容的功能新增
- **修订号**：向下兼容的问题修复

当前阶段：**0.x.y**（预发布，可能有不兼容变更）

---

## 贡献流程

详见 CONTRIBUTING.md。简而言之：

1. Fork 仓库
2. 创建特性分支（`feat/xxx` 或 `fix/xxx`）
3. 提交变更（遵循 Conventional Commits）
4. 确保测试通过
5. 提交 PR
6. 等待审查和合并
