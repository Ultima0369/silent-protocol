# neca2 改进建议

> 基于对 neca2 当前源码（2026-05-18）的深度分析，提出 7 条改进建议。
> 每条包括：当前问题 → 改进方案 → 预期收益。

---

## 1. 编解码器抽象层可扩展性不足

**当前问题：**
`JsonCodec` 实现了 `Codec` 接口，但目前只有一个实现。当未来需要切换二进制编解码（protobuf/msgpack）时，需要改 `relay.ts` 和 `router.ts` 中硬编码的 `JsonCodec` 实例化逻辑。

**改进方案：**
- 在 `protocol/codec.ts` 中增加 `CodecFactory`，根据配置（`config.codec` 字段）返回对应实现
- 注册模式：`CodecFactory.register('json', () => new JsonCodec())`
- 使用方只依赖 `Codec` 接口，不依赖具体实现

**预期收益：**
- 切换编解码方式只需改一行配置
- 第三方可以注册自定义编解码器，扩展性大幅提升

---

## 2. 会话持久化目前是手动调用

**当前问题：**
`session.ts` 中 `persist()` 需要业务方手动调用。这意味着如果某条消息的处理流程在持久化之前崩溃，会话状态会丢失。

**改进方案：**
- 在 `SessionManager` 中增加自动持久化选项（`autoPersist: boolean`）
- 每次 `create()` 和 `transition()` 时自动写盘
- 增加 debounce 机制（100ms），避免高频状态变更时频繁 IO

**预期收益：**
- 机器崩溃后恢复时，最多丢失 100ms 内的状态
- 业务方不需要记得调 `persist()`

---

## 3. 缺少消息重试与去重机制

**当前问题：**
`SessionManager` 记录了 `retryCount`，但没有自动重试逻辑。消息发送失败不会自动重传。同时，接收方如果收到重复消息（网络重传导致），也没有去重处理。

**改进方案：**
- 在 `relay.ts` 中增加一个 `RetryQueue` 组件：
  - 扫描状态为 `sent` 但超过超时时间未 `ack` 的消息
  - 自动重新编码并投递（最多重试 3 次，指数退避）
  - 消息 ID 不变，接收方可根据消息 ID 去重

**预期收益：**
- 网络抖动时消息不会丢失
- 不会因为重传导致命令被执行两次

---

## 4. 日志与可观测性不足

**当前问题：**
目前 neca2 的日志只输出到 `stderr`，没有结构化日志、没有日志级别区分、没有与 neca 的 `blackboard` 集成。

**改进方案：**
- 引入结构化日志模块 `src/logger.ts`，输出 JSON Lines 到 `~/.neca/neca2.log`
- 日志级别：`debug | info | warn | error`
- 每个消息的流转过程记录完整 trace：`id`, `from`, `to`, `type`, `status`, `duration`
- 集成到 `neca2_health` 和 `neca2_relay_status` 的返回值中

**预期收益：**
- 调试时可以用 `jq` 过滤日志
- 用户可以看到消息流转的完整轨迹
- 问题排查时间缩短 50%+

---

## 5. 缺少协议合规性自检工具

**当前问题：**
协议规范定义了 9 种消息类型、编解码规则、路由逻辑。但如果一个第三方实现了自己的网关，如何验证它的实现与规范一致？目前没有标准测试套件。

**改进方案：**
- 新建 `tests/compliance/` 目录
- 编写一套协议合规性测试（PCT: Protocol Compliance Tests）
- 每种消息类型至少包含：编码→解码→字段校验→边界条件
- 提供一个 CLI 命令 `neca2 compliance` 运行全套测试

**预期收益：**
- 任何语言的第三方实现都可以运行 PCT 来验证兼容性
- 规范改动时，PCT 确保所有实现同步更新

---

## 6. 会话管理器不支持批量过期清理

**当前问题：**
`cleanup()` 方法只根据单次调用清理，没有定期后台任务。长时间运行的 neca2 进程会积累大量已完成/超时的会话，导致内存泄漏。

**改进方案：**
- 在 `SessionManager` 中增加一个定时器（默认每 5 分钟执行一次）
- 清理策略：
  - `completed` 状态超过 10 分钟 → 删除
  - `failed` 状态超过 30 分钟 → 删除
  - `pending` 状态超过 2 小时 → 标记失败并删除
- 可配置：`sessionTTL`、`cleanupInterval`

**预期收益：**
- 长期运行的 neca2 进程内存不会无限增长
- 磁盘上的持久化文件也不会无限堆积

---

## 7. 缺少多模型路由的优先级调度

**当前问题：**
当前 `router.ts` 只根据 `to` 字段做简单消息路由。如果未来有多个消费者（比如同时有 cloud_claude 和 cloud_gemini），没有优先级和负载均衡逻辑。

**改进方案：**
- 在 `router.ts` 中增加可插拔的路由策略：
  - `RoundRobinRouter` — 轮流分发
  - `PriorityRouter` — 根据消息的 `priority` 字段选择节点
  - `LeastLoadedRouter` — 根据节点当前任务数分发
- 注册模式：`router.register('cloud_claude', handler, { weight: 3 })`

**预期收益：**
- neca2 可以对接多个云端模型供应商
- 高优先级任务可以插队
- 负载低的节点能被优先调度

---

## 改进路线图优先级

| 优先级 | 改进项目 | 难度 | 收益 |
|--------|---------|------|------|
| P0 | 自动持久化 + 恢复 | 低 | 🔥 数据安全 |
| P0 | 日志与可观测性 | 中 | 🔥 可调试性 |
| P1 | 消息重试与去重 | 中 | ⚡ 可靠性 |
| P1 | 批量过期清理 | 低 | ⚡ 内存安全 |
| P2 | CodecFactory 可扩展性 | 低 | 🧩 扩展性 |
| P2 | 协议合规性自检 | 中 | 🧩 生态兼容 |
| P3 | 多模型优先级调度 | 高 | 🚀 规模扩展 |
