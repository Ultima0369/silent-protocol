## 一·架构全景

Silent Protocol 定义了一个四层协作模型，四个参与者各司其职：

```
┌──────────────────────────────────────────────────────────────┐
│                    人类（产品经理）                            │
│  角色：提出需求、品尝成果、给予反馈                           │
│  语言：自然语言                                               │
│  接口：Chatbox / 任何支持 API 的客户端                        │
│  特征：低频率、高影响、尝菜式反馈                             │
└───────────────────────┬──────────────────────────────────────┘
                        │ API
                        ▼
┌──────────────────────────────────────────────────────────────┐
│                   云端 DeepSeek（翻译官）                      │
│  角色：理解人类需求、拆解为子任务、派发给执行层                │
│  语言：自然语言 ↔ 紧凑协议 双向翻译                           │
│  接口：DeepSeek API + neca MCP 工具集                         │
│  特征：高推理深度、跨学科知识覆盖、任务编排                   │
└───────────────────────┬──────────────────────────────────────┘
                        │ neca MCP · 紧凑协议
                        ▼

│               neca 网关（通信枢纽）                           │
```

## 二·角色职责详解

### 2.1 人类（产品经理）

**不是**：技术实现者、bug 修复者、文档写手。
**而是**：需求的提出者、成果的检验者、方向的修正者。

人类的工作方式：
1. 用自然语言描述目标（"我想要一个自动整理桌面文件的脚本"）
2. 等待智能体们协作产出
3. 检验成果并给出反馈（"这个功能不错，但速度太慢了"）
4. 重复直到满意

**关键约束**：人类不在智能体执行过程中被拉入技术细节。
这是架构设计的硬要求——任何需要人类介入技术细节的流程都是设计缺陷。

### 2.2 云端 DeepSeek（翻译官）

**不是**：执行者、代码生成器（那是本地 Claude 的事）。
**而是**：人类与硅基网络之间的桥梁。

核心职责：
1. **需求理解**：将人类的自然语言（可能模糊、不完整、有歧义）解析为结构化的任务描述。
2. **任务拆解**：将复合需求分解为可独立执行的子任务。
3. **智能派发**：根据任务性质判断由谁执行（本地命令？云端查询？）。
4. **质量把关**：汇总各子任务的结果，判断是否达到人类要求。
5. **反馈翻译**：将人类的尝菜反馈重新拆解为修正任务。

### 2.3 本地 Claude Code（总工程师）

**不是**：翻译官、任务拆解器。
**而是**：在本地机器上执行操作的手和脚。

核心职责：
1. **执行命令**：编译、运行、测试、部署。
2. **文件操作**：读写、搜索、修改代码文件。
3. **本地工具调用**：git、node、编译器、测试框架。
4. **向云端求助**：遇到需要专业知识或复杂推理的任务时，通过 neca relay 向云端 Claude 查询。

### 2.4 云端 Claude API（顾问团）

**不是**：执行者、文件操作者。
**而是**：知识库和推理引擎。

核心职责：
1. **知识问答**：提供专业技术知识的查询服务。
2. **复杂推理**：处理本地 Claude 无法独立完成的逻辑分析。
3. **代码审查**：对本地产生的代码进行质量评估。
4. **方案建议**：在多个可行方案中推荐最优解。

## 三·通信流详解

### 3.1 标准工作流（8步）

```
Step 1: 人类 → 云端 DeepSeek
  "帮我优化一下麻将项目的牌型判断，太慢了。"
  语言：自然语言
  传输：Chatbox → API

Step 2: 云端 DeepSeek 内部处理
  拆解为子任务：
  - T1: 分析当前牌型判断的性能瓶颈
  - T2: 搜索最优算法方案
  - T3: 重构代码
  - T4: 跑测试验证
  语言：内部思维链（自然语言/逻辑推理）
  
Step 3: 云端 DeepSeek → 本地 Claude Code（通过 neca）
  消息类型：exec
  载荷：{ cmd: "cd C:\\majiong && python analyze.py" }
  协议：紧凑协议（JSON）
  
Step 4: 本地 Claude Code 执行
  运行性能分析工具
  发现问题：递归回溯算法复杂度 O(2^n)
  
Step 5: 本地 Claude Code → 云端 Claude API（通过 neca relay）
  消息类型：query
  载荷：{ question: "麻将牌型判断最优算法？", context: "..." }
  协议：紧凑协议
  
Step 6: 云端 Claude API → 本地 Claude Code（通过 neca relay）
  消息类型：report
  载荷：{ taskId: "...", result: "推荐位掩码查表法" }
  协议：紧凑协议

Step 7: 本地 Claude Code 根据建议修改代码
  完成 T3、T4
  
Step 8: 本地 Claude Code → 云端 DeepSeek（通过 neca）
  消息类型：report
  载荷：{ taskId: "...", status: "completed", result: "优化完成，12ms→3ms" }
  协议：紧凑协议
  
  → 云端 DeepSeek 汇总 → 自然语言回复人类：
  "搞定了。用了位掩码查表法，性能从 12ms 降到 3ms。"
```

### 3.2 尝菜式反馈流

```
Round 1:
  人类 → 云端 DeepSeek: "做个网页版麻将计分器"
  → 内部协作 → 产出 v1
  云端 DeepSeek → 人类: "做好了，在这里: ..."

Round 2:
  人类 → 云端 DeepSeek: "界面不太好看，按钮太大了"
  → 内部协作 → 产出 v2
  云端 DeepSeek → 人类: "调整了按钮大小，加了一点配色"

Round 3:
  人类 → 云端 DeepSeek: "对了，再加个音效"
  → 内部协作 → 产出 v3
  云端 DeepSeek → 人类: "加了碰牌和胡牌的音效"
```

人类只参与 Round 的开头和结尾。Round 中间的硅基协作完全对用户透明。

## 四·协议栈层次

```
层次 4: 人类接口层
  协议：自然语言
  组件：Chatbox / 自定义客户端
  责任：人与硅基网络之间的界面

层次 3: 翻译层
  协议：自然语言 + 结构化指令
  组件：云端 DeepSeek
  责任：将模糊需求翻译为精确任务描述

层次 2: 通信层（核心）
  协议：紧凑协议（JSON / 未来二进制）
  组件：neca 网关（protocol/ + relay/）
  责任：消息路由、协议转译、会话管理

层次 1: 执行层
  协议：操作系统 API / 本地工具
  组件：本地 Claude Code
  责任：执行具体操作

层次 0: 知识层
  协议：LLM API
  组件：云端 Claude API
  责任：提供深度知识和推理能力
```

## 五·关键设计决策

### 5.1 为什么 neca 是网关而不是总线？

总线架构允许任意实体互相通信，灵活性高但安全性差。
网关架构要求所有消息经过 neca 路由，原因：

1. **安全审计**：所有通信有统一记录点。
2. **白名单执行**：neca 在路由时检查目标是否合法。
3. **协议转译**：未来不同实体可能使用不同版本的协议，neca 负责转译。
4. **会话管理**：统一处理超时、重试、去重。

### 5.2 为什么云端 DS 不直接调用 Claude API？

技术上是可行的，但违背了分层原则：

- 云端 DS 调用 Claude API 需要云端 DS 自己去管理另一个 API 连接。
- 让 neca 统一管理所有外部连接，降低了云端 DS 的复杂度。
- 通过 neca relay，所有通信路径对系统有统一的可见性。

### 5.3 为什么不把所有逻辑放在本地？

某些任务需要：
- 大规模知识库（本地无法存储全部训练数据）。
- 高算力深度推理（本地硬件有限）。
- 模型多样性带来的认知红利（不同模型有不同的盲区）。

## 六·部署模型

### 6.1 最小部署（单机）

```
[Chatbox + neca + Claude Code] 在同一台机器上
云端 DS 和云端 Claude 通过 API 访问
```

### 6.2 标准部署（开发环境）

```
[Chatbox] → [neca + Claude Code] 在开发机上
云端 DS 和云端 Claude 通过 API 访问
```

### 6.3 扩展部署（团队协作）

```
[Chatbox A] ─┐
[Chatbox B] ─┼──→ [neca 服务端 + Claude Code 集群]
[Chatbox C] ─┘
云端 DS 和云端 Claude 通过 API 访问
```

## 七·与现有技术的对比

| 维度 | MCP (Anthropic) | A2A (Google) | CrewAI | Silent Protocol |
|------|----------------|-------------|--------|-----------------|
| 通信格式 | JSON-RPC | JSON + 自然语言 | 自然语言 | 紧凑协议 |
| 路由 | 无（直连） | Agent Card 发现 | 固定拓扑 | 网关路由 |
| 人类角色 | 调用者 | 启动者 | 不参与 | 尝菜者 |
| 跨模型 | 否 | 是 | 否 | 是 |
| 跨云/本地 | 否 | 是 | 否 | 是 |
| 二进制预留 | 无 | 无 | 无 | 是 |

## 八·未来扩展

1. **多人类支持**：允许多个"产品经理"同时参与，通过 neca 协调反馈冲突。
2. **任务优先级调度**：高优先级任务插队。
3. **统计与仪表盘**：记录每个实体的 token 消耗、任务耗时、成功率。
4. **插件系统**：第三方开发者可以注册新的消息类型和处理 handler。
5. **联邦 neca**：多个 neca 网关互联，形成更大的协作网络。
