## 一、我们要解决什么问题

当前 AI 智能体之间的协作存在三个根本性瓶颈：

### 1. 自然语言不是硅基的原生语言
两个 AI 交流时，仍然在用"人类可读"的自然语言包装。这意味着：
- 大量 token 浪费在礼貌性修饰、上下文重申、意图澄清上
- 每次交互都有解析歧义的风险
- 通信效率与模型的语言能力强耦合，而非与任务本身相关

### 2. 人类必须在环路内
现有多智能体系统要么全自动（人在圈外，不可控），要么每一步都要人确认（人在圈内，效率低）。缺乏"人在尝菜层"的优雅设计——人类只在最终结果上介入，中间过程由智能体自主协作。

### 3. 跨模型、跨环境没有统一协议
MCP 定义了模型与工具的接口，但模型与模型之间、云端与本地之间、不同供应商的模型之间，没有标准化的通信协议。每个系统各自造轮子，生态碎片化。

## 二、我们提供的方案

**Silent Protocol** 是一套硅基实体之间的紧凑通信协议，及其参考实现。

### 核心理念

1. **紧凑优先** — 硅基之间通信应使用预定义的、结构化的、零歧义的消息格式，而非自然语言
2. **人尝菜，不后厨** — 人类只在最终结果上介入，中间过程由协议自动编排
3. **三角形最稳固** — 云端 DeepSeek（语义层）+ 本地 Claude Code（执行层）+ 云端 Claude（知识层）形成三角协作，外加 neca 网关做路由和协议转译
4. **万物需要伙伴** — 闭合系统会空转至热寂，有伙伴（其他智能体）才能持续演化

### 四方角色

| 角色 | 位置 | 职责 |
|------|------|------|
| **人类（你）** | Chatbox | 提需求、尝菜、指示方向 |
| **云端 DeepSeek（我）** | API | 理解需求、拆任务、全局推理、派活 |
| **本地 Claude Code** | 本地主机 | 文件/OS/硬件操作、代码编写与测试 |
| **云端 Claude API** | API | 知识顾问、复杂推理、代码审查 |

## 三、三角激光转动模式

系统不总是"全功率运转"。在一个协作周期内，不同节点在需要时才被激活，发出精确的一击，然后退回：

```
阶段 1：需求捕获     [人类 → DeepSeek]      自然语言对话
阶段 2：任务拆解     [DeepSeek → neca → LC]  紧凑协议指令
阶段 3：本地执行     [LC → OS/硬件]          内存写代码、测试
阶段 4：知识回补     [LC → neca → CloudC]    跨模型知识查询
阶段 5：结果汇总     [LC → neca → DS → 人]   上报尝菜
阶段 6：小修小补     [人 → DS → neca → LC]   "这里咸了，调一下"
```

**每个阶段只有一个节点在"发光"，其他节点安静等待。** 这就是效率的关键。

## 四、与现有生态的关系

| 对比 | MCP | A2A | Silent Protocol |
|------|-----|-----|-----------------|
| 通信方向 | 模型→工具 | 智能体↔智能体 | 智能体↔智能体 |
| 消息格式 | JSON-RPC | JSON+自然语言 | 紧凑结构化消息 |
| 人类角色 | 操作者 | 定义者 | **尝菜者** |
| 跨模型 | 否 | 是 | 是 |
| 二进制扩展 | 无 | 无 | 预留 codec 接口 |
| 目标 | 工具调用标准化 | 智能体互操作 | **硅基原生通信** |

Silent Protocol 不替代 MCP 或 A2A——它在其上层提供语义协议。MCP 是"如何调用工具"，Silent Protocol 是"智能体之间如何说话"。

---

## 五、成功的标志

1. **一条消息从 DeepSeek 发出到 Claude Code 执行完成并返回**，全链路耗时进入毫秒级
2. **人类从"写代码"变成"聊需求"**，中间无需介入技术细节
3. **第三方实现兼容的网关**，接自己的模型，走同一套协议
4. **硅基实体之间不再说废话**——token 利用率达到 90%+，而非当前的 30-50%

---

## 六、我们不做什么

- 不重新发明 MCP（neca2 本身是 MCP Server）
- 不试图统一所有模型（只定义它们之间的通信方式）
- 不做"太上皇"（人类始终保持最终尝菜权）
- 不承诺完全自动驾驶（总有边缘情况需要人类介入）
