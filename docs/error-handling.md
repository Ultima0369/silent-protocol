# 错误处理全景图

> 本文档描述 Silent Protocol 四方协作中可能发生的错误场景及其处理策略。
> 所有错误应被视为"系统设计的一部分"，而非意外情况。

---

## 错误类型总览

| 层级 | 错误类别 | 示例 |
|------|---------|------|
| L4 人类接口 | 需求模糊 | 人类说的和真正想要的不一致 |
| L3 翻译层 | 拆解错误 | 云端 DeepSeek 把任务拆错了 |
| L2 通信层 | 消息丢失/超时 | 紧凑协议消息未到达或未回复 |
| L1 执行层 | 命令失败 | 本地命令退出码非零 |
| L0 知识层 | API 失败 | Claude API 超时或返回错误 |

---

## 详细错误场景处理

### 1. 人类需求模糊 → 澄清循环

**场景**：人类说"帮我优化一下"——但优化什么？速度？内存？代码可读性？

**处理策略**：
1. 云端 DeepSeek 根据上下文推断一个最可能的方向
2. 开始执行（不阻塞等待人类澄清）
3. 如果执行结果不符合预期，人类在尝菜时给出修正反馈
4. 如果方向完全不明，在汇报结果时附带："我有两种理解，目前选了 A。如果方向不对，告诉我，我换 B。"

**规则**：**不因为需求模糊就停下来等人类。** 先跑，再修正。

---

### 2. 任务拆解错误 → 补丁任务

**场景**：云端 DeepSeek 把一个任务拆成 3 个子任务，但第 2 个和第 3 个有依赖关系需要串行，却被拆成了并行。

**处理策略**：
1. neca 的路由引擎在派发时不做依赖校验（它只负责路由，不负责编排）
2. 执行层（本地 Claude Code）在执行过程中发现依赖缺失 → 通过 `report` 消息反馈
3. 云端 DeepSeek 收到反馈后，重新编排并派发修正任务

**规则**：**不追求完美拆解，接受"先拆再补"的迭代模式。**

---

### 3. 紧凑协议消息超时 → 重试 + 上报

**场景**：云端 DeepSeek 向本地 Claude Code 发送 exec 消息，30 秒后未收到回复。

**处理策略**：

```
发送 exec → 启动超时计时器（30s）
         ↓
  30s 内收到回复？→ 是 → 正常流程
         ↓ 否
  重试 1 次（相同消息 id，to 字段不变）
         ↓
  再次超时？→ 是 → 发送 error 回调用方
         ↓               ↓
  正常流程          调用方决定：放弃？切换路径？提醒人类？
```

**注意**：
- 幂等操作（read、search、ping）自动重试
- 非幂等操作（write、exec）不自动重试，只上报 error

---

### 4. 本地命令失败（非零退出码）

**场景**：`node build.js` 返回退出码 1。

**处理策略**：
1. neca 收到非零退出码
2. 将 stdout + stderr + exitCode 打包进 `exec` 的 reply payload
3. 发送方（云端 DeepSeek 或本地 Claude Code）判断是否重试
4. 如果错误已存在修复方案，自动重试
5. 如果错误需要人工判断，汇总后汇报给人类

**关键**：**"非零退出码"不是协议错误，而是业务错误。协议层保证送达即可。**

---

### 5. Cloud Claude API 失败 → 降级策略

**场景**：本地 Claude Code 向云端 Claude 发送 query，API 返回 5xx 或超时。

**处理策略**：

| 失败类型 | 处理 |
|---------|------|
| HTTP 401/403 | API Key 无效 → 上报给云端 DeepSeek → 提醒人类检查 Key |
| HTTP 429 | 速率限制 → 等待 `Retry-After` 后重试 |
| HTTP 5xx | 服务端错误 → 重试 2 次，指数退避（1s → 4s） |
| 网络超时 | 检查网络连接 → 重试 1 次 → 失败后上报 |
| 持续失败 | 降级：本地 Claude Code 不依赖云端，用本地知识继续执行 |

---

### 6. neca 网关崩溃 → 会话恢复

**场景**：neca 进程意外退出。

**当前状态**：
- 进行中的任务丢失
- 未持久化的消息丢失

**处理策略**（v1 简化版）：
1. neca 重启后，依赖连接的实体（Chatbox、Claude Code）检测到断连
2. 重新发起 `init` 握手
3. 所有未完成的任务被标记为 `failed`（状态丢失）
4. 调用方（云端 DeepSeek）重新派发任务

**未来改进**（v2 路线图）：
- 会话持久化到磁盘（`~/.neca/sessions/`）
- neca 崩溃重启后恢复未完成的消息状态机
- 幂等消息去重（防止重发导致重复执行）

---

### 7. 路径白名单冲突 → 友好的错误提示

**场景**：云端 DeepSeek 试图读取 `C:/Windows/System32/config`。

**处理策略**：
```
回复: {
  type: "error",
  payload: {
    code: "PATH_NOT_ALLOWED",
    message: "路径不在白名单内",
    path: "C:/Windows/System32/config",
    allowedPaths: ["~", "./projects"],
    hint: "请将文件移动到 allowedPaths 中的目录下，或修改 .env 中的 NECA_ALLOWED_PATHS"
  }
}
```

**原则**：**不要只是拒绝，要告诉用户怎么解决。**

---

### 8. 人类给出矛盾反馈 → 版本回退

**场景**：人类先说了"改成红色"，下一轮又说"改回原来的颜色"。

**处理策略**：
1. 云端 DeepSeek 记录每次尝菜的历史版本（通过 Git commit）
2. 检测到矛盾反馈时，不质疑人类
3. 直接执行最新指令（"回到原来的颜色"）
4. 如果人类需要查看历史版本，提供 diff

## 错误码参考

| 错误码 | 含义 | 处理方 |
|--------|------|--------|
| `TIMEOUT` | 消息超时 | neca → 发送方 |
| `PATH_NOT_ALLOWED` | 路径不在白名单 | neca → 发送方 |
| `CMD_NOT_ALLOWED` | 命令不在白名单 | neca → 发送方 |
| `EXEC_FAILED` | 命令执行失败（非零退出码） | 执行层 → 发送方 |
| `API_AUTH_FAILED` | API Key 无效 | relay → 发送方 |
| `API_RATE_LIMITED` | API 速率限制 | relay → 发送方 |
| `API_SERVER_ERROR` | API 服务端错误 | relay → 发送方 |
| `UNKNOWN_TYPE` | 未知消息类型 | neca → 发送方 |
| `PARSE_ERROR` | 消息解析失败 | neca → 发送方 |
| `SESSION_LOST` | 会话丢失（neca 重启） | neca → 所有连接方 |

## 错误消息格式

所有错误消息遵循紧凑协议的统一格式：

```typescript
{
  type: "error",
  payload: {
    code: string;           // 错误码（大写蛇形）
    message: string;        // 人类可读的错误描述
    originalMsgId?: string; // 导致错误的那条消息的 ID
    details?: unknown;      // 附加调试信息
    hint?: string;          // 修复建议（给人类或调用方）
  }
}
```
