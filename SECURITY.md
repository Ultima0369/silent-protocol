# 安全政策

> Silent Protocol 涉及跨网络命令执行和文件读写操作。
> 安全不是附加功能，是核心设计约束。

---

| 版本 | 支持状态 |
|------|---------|
| v1.x | ✅ 活跃支持 |
| < v1 | ❌ 不受支持 |

## 安全架构概述

Silent Protocol 的安全性建立在以下设计原则上：

### 1. 网关路由

所有通信必须经过 neca 网关，不允许多个实体之间直接连接。这确保了：
- 所有通信路径可审计
- 统一的安全策略执行点
- 无隐形的"后门"连接

### 2. 分层权限

| 层 | 权限 | 约束 |
|----|------|------|
| 人类 | 完整 | 无限制（但受客户端安全措施约束） |
| 云端 DeepSeek | 任务派发 | 只能通过 neca MCP 工具调用，受工具白名单约束 |
| 本地 Claude Code | 文件读写+命令执行 | 受路径白名单和命令白名单双重约束 |
| 云端 Claude API | 纯推理 | 无文件系统权限，无命令执行权限 |

### 3. 白名单约束

- **路径白名单**：文件操作限制在用户显式授权的目录内
- **命令白名单**：shell 执行限制在安全命令集内
- **速率限制**：防止滥用和 DoS 攻击

### 4. 紧凑协议的安全优势

紧凑协议的预定义消息类型天然限制了攻击面：
- 无法注入自然语言指令
- 消息类型和载荷结构严格预定义
- 类型校验在解析层完成，不交给执行层判断

## 已知安全边界

以下情况不在 Silent Protocol 的安全承诺范围内：

1. **Chatbox 客户端本身**：如果 Chatbox 被攻破，攻击者可以冒充人类用户
2. **API Key 泄露**：DeepSeek 或 Claude 的 API Key 泄露等于攻破了身份认证
3. **主机系统被攻破**：如果运行 neca 的机器本身被 rootkit 感染，所有白名单约束可被绕过
4. **网络层攻击**：MITM 攻击可能截获或篡改 API 通信（建议使用 HTTPS/TLS）

## 安全最佳实践

### 给用户

```bash
# ❌ 不要这样配置
NECA_ALLOWED_PATHS=/

# ✅ 应该这样配置
NECA_ALLOWED_PATHS=~,./projects
```

### 给开发者

- 所有用户输入必须经过校验和转义
- exec 命令必须拆分检查白名单，不信任输入的完整字符串
- 文件路径必须解析为绝对路径后检查白名单
- 永远不要在日志中记录 API Key 或令牌

### 给部署者

- 使用环境变量而非配置文件存储密钥
- 在 .env 文件中设置严格的文件权限（0400）
- 不要在共享服务器上运行 neca
- 定期轮换 API Key

## 安全相关配置参考

| 配置项 | 安全意义 |
|--------|---------|
| `NECA_ALLOWED_PATHS` | 限制文件系统访问范围 |
| `NECA_ALLOWED_COMMANDS` | 限制 shell 命令执行范围 |
| `RATE_LIMIT_PER_TYPE` | 防止单类型消息滥用 |
| `RATE_LIMIT_TOTAL` | 防止全局 DoS |
| `PROTOCOL_CODEC` | 二进制编码器可减少侧信道信息泄露 |
